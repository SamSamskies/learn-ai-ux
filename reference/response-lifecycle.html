The Response Lifecycle

The states every nondeterministic reply moves through, and the deliberate UI treatment each one earns.

The states

StateWhat's happeningUI treatmentAI SDK signal
idle No request in flight. Awaiting input. Composer enabled. Empty state or prior conversation. Suggest a gentle first step. status === 'ready' & no messages
submitted Request sent; no stream yet. Acknowledge instantly: echo the user message, disable resend, show a "thinking" affordance. status === 'submitted'
thinking Stream open (metadata/latency) but no tokens yet. Distinct "thinking" indicator — not the same as typing. This is the gap where trust leaks. streaming + last assistant msg has empty parts
streaming Tokens arriving. Render progressively. Show a Stop button. Keep the composer aware a reply is in progress. streaming + non-empty parts
complete Full reply received. Re-enable composer. Offer follow-up affordances (copy, retry, feedback, sources). status === 'ready' after a reply
stopped User aborted mid-stream. Keep partial output, mark it as stopped, offer "continue" / "regenerate". stop() → returns to ready (abort ignored)
error Request failed (network, provider, rate limit). Plain-language message, a retry, and preserve the user's input. Never a raw stack trace. status === 'error', error object set
The one that bites

The AI SDK flips status to streaming the moment the connection opens — including metadata — before any text. If you treat streaming as "typing," users see a silent, misleading state during the thinking gap. Detect it by checking the last assistant message's parts are still empty.

Design principles

Acknowledge instantlyPerceived latency is set in the first 100ms. Echo input and show a state before the model does anything.
Make progress legibleStream steps, not just tokens where you can — "Searching sources → Reading → Writing" beats an opaque spinner.
Every exit is designedstopped and error are first-class states, not afterthoughts. Preserve user work in both.
Strategic frictionWhere a wrong action is costly, a deliberate confirm step builds more trust than raw speed.
