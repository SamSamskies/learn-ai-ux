Perceived Latency

How slow an interaction feels — governed by acknowledgement and legible progress, not raw milliseconds.

The three levers

LeverWhat it doesWhen to use
Acknowledge instantly Echo the user's input and disable resend the moment they hit Send — before the server answers. Every send. Non-negotiable.
Legible progress Replace generic spinners with named steps as soon as you know what's happening ("Searching documents…"). Any gap longer than ~300 ms. Especially the thinking gap.
Strategic friction Deliberately slow or gate an action (confirm, review) to build trust or prevent harm. Destructive actions, high-stakes outputs — not the default wait state.

In-flight micro-states

During a single turn, the user experiences three distinct waits. Collapsing them into one spinner hides progress and feels slower than it is.

Micro-stateWhat's happeningUI treatmentSignal
submitted Request sent; HTTP not connected yet. Lightweight ack — "Got it." No heavy skeleton. status === 'submitted'
thinking Stream open but no app progress yet — metadata, first byte, or empty assistant message. Distinct thinking indicator — not the same as active search. streaming + no research snapshot / no step labels
working Backend work is underway with known steps. Named step labels + optional skeleton for the expected output shape. Research phase + searched / searchedDocs flags

Common mistakes

One spinner for everything"Working…" during submitted promises progress you don't have yet.
Silent thinking gapAssistant message exists but no data-research part yet → user sees only their bubble. Worst case.
Typing cursor before tokensRendering streaming UI on status alone — connection ≠ content.
No scroll on sendUser's question scrolls off-screen; the wait feels like nothing happened.
