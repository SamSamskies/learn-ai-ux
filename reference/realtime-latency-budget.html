Realtime Latency Budget

Conversational voice breaks around ~500–800 ms from end-of-speech to first model audio. Budget the stack, measure each segment, tune VAD — don't guess.

The additive stack

Unlike text (one HTTP round-trip), voice latency is sum of fixed segments you partially control:

User stops talking
  └─ VAD tail (silence_duration_ms)     ← you configure
  └─ speech_stopped event
  └─ Model thinking gap                 ← show "Thinking…"
  └─ First output audio / transcript    ← start of "Speaking…"

Your thinking phase covers the model gap. VAD tail happens before that — users feel it as "why isn't it responding yet?" after they finished their sentence.

First audio — — Phase → speaking; latency timer stops here

server_vad knobs

KnobDefaultEffectWhen to change
silence_duration_ms 500 Silence before speech_stopped — direct add to perceived wait Feels sluggish → lower (400). Cuts you off mid-thought → raise (700–900)
threshold 0.5 How loud input must be to count as speech (0–1) Noisy room / false barge-in → raise (0.7–0.85). Quiet mic → lower
prefix_padding_ms 300 Audio included before detected speech start Clipped first syllables → raise slightly; rarely the first dial to turn
interrupt_response true Auto-cancel model when user speaks Set false only for HITL / moderation — disables natural barge-in

Measure in the probe

speech_stopped  →  stamp Date.now()
first audio delta  →  latencyMs = now - stamp

Log speech_stopped → first audio — that is end-to-end turn latency the user feels. Compare before/after VAD changes. Green under ~800 ms; investigate above.

semantic_vad (alternative)

type: 'semantic_vad' uses the model to guess utterance completion instead of fixed silence. Tradeoff: smarter turn boundaries vs less predictable timing. Tune with eagerness (low / medium / high). Start with server_vad until silence-based detection is clearly wrong. Full cheat sheet: semantic VAD reference.
