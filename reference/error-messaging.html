Error Messaging

Two error channels, one set of user-facing rules — plain language, preserved context, retry in place.

Two channels

ChannelOriginSignal in AI SDKWho writes the copy
Domain Server completed the stream but reported failure in app state (e.g. phase: 'error' in data-research) Inside message parts Your API / stream handler — set research.error server-side
Transport HTTP, network, auth, or SDK parse failure before/during delivery Top-level error from useChat Client mapper (friendlyTransportError) — never show error.message raw

Three rules (both channels)

Plain languageSpecific enough to act on; blame-free; no status codes or stack traces in the UI.
Preserve contextKeep the user's question and any partial output on screen.
Retry in placeregenerate({ messageId }) when an assistant message exists. Re-send question text only when failure happened before any assistant message was created.

Common transport mappings

Raw signalUser sees
Failed to fetch, network errorsConnection problem — check your network and try again.
401 / 403 / unauthorizedYou're not signed in. Refresh the page and try again.
429 / rate limitToo many requests — wait a moment and try again.
500 / 502 / 503Our servers hit a snag. Try again in a few seconds.
AbortErrorThe request was cancelled. (Usually hidden — user already stopped.)
Anything elseSomething went wrong. Please try again.
Dev vs prod

Log error.message (and stack) to the console or your observability tool in development. The UI always gets the mapped string.
