Multimodal Voice Output

Generate images from a Realtime voice session via a client-owned function tool — confirm the prompt on screen, then show the picture while speech narrates.

Pipeline

  1. User asks aloud to draw / sketch / visualize.
  2. Model emits function_call for generate_illustration with a prompt arg.
  3. Client holds — dock: "Confirm the image prompt…"; editable textarea seeded from args.
  4. User edits (optional) → Generate, or Cancel.
  5. On Generate: dock "Generating image…" → POST /api/voice/generate-image → append image turn to transcript.
  6. function_call_output + response.create → model speaks a one-line caption.

Why gate + edit

RiskWhy it matters in voiceUX
STT error Misheard nouns ruin creative prompts Editable prompt before spend
Cost / latency Image gen is slower and pricier than a lookup Explicit confirm; cheap model settings for prototypes
Wrong modality User may have meant "explain," not "draw" Cancel completes the loop; model stays in voice

Images API (server)

await openai.images.generate({
  model: "gpt-image-1-mini",
  prompt,
  size: "1024x1024",
  quality: "low",
});
// result.data[0].b64_json → data:image/png;base64,...

Keep the key on the server. Client only receives a data URL (or a short-lived hosted URL if you add storage later).

Transcript turn

{
  role: "assistant",
  text: "edited prompt used for generation",
  imageUrl: "data:image/png;base64,…",
}

Render imageUrl as an <img>; use text as caption/alt. Voice says the headline; the screen holds the artifact.

Tool risk split (voice)

ToolGate?Label

stage_research_brief Confirm (show query) Waiting for your approval…
generate_illustration Confirm + edit prompt Confirm… → Generating image…

Out of scope (v1)

  • Variations grid (Shape of AI Variations) — one image per call
  • Streaming partial images
  • Persisting images beyond the tab session
  • Feeding the generated image back into Realtime as input_image unless you explicitly need follow-up vision
