Edit-Before-Send

Citations are claim-level now. The next human-in-the-loop win is before the run — let AI sharpen a rough question, but keep the user in the driver's seat until they hit Send.

Where you actually are

Your composer is honest: what you type is what gets researched. That's good for agency, but vague questions waste an expensive multi-tool run:

Gap — no prompt assistance at the front door

"competitors" or "is this a good idea" goes straight to web search + structured output. The model guesses scope. Users can't see or fix a rewrite because there isn't one — they only discover the mismatch after the brief lands.

The Shape of AI prompt enhancer pattern brings hidden prompt rewriting into the open: suggest a clearer question, drop it in the composer, let the user edit, then send. That's edit-before-send — human-in-the-loop on the instruction itself.

The principle

Enhance explicitlyLabel the action ("Refine") and the result ("Refined — review before sending")
Never auto-sendRefined text replaces the draft in the textarea; Send stays manual
Undo beats retypeOne click restores the original draft if the rewrite drifts
Lightweight callFast model, no tools — this is cheap intent-shaping, not a full research run
Primary source (skim) Shape of AI · Prompt enhancer — make enhancement explicit and reviewable, preserve user intent, require just enough input before enhancing. See also edit-before-send reference.

Build it (~15 min)

  1. Create lib/refine-prompt.ts. A small server helper — one job, no tools:

    import "server-only";
import { openai } from "@ai-sdk/openai";
import { generateText } from "ai";

const INSTRUCTIONS =
  "Improve research questions for a solo builder. Rewrite the draft into one clear, specific question. Preserve topic and constraints. Do not invent facts. Return only the improved question.";

export async function refinePrompt(draft: string): Promise<string> {
  const { text } = await generateText({
    model: openai("gpt-4o-mini"),
    system: INSTRUCTIONS,
    prompt: draft,
  });
  return text.trim();
}

  2. Add app/api/refine-prompt/route.ts. Reuse your existing guard — same auth/rate-limit as research:

    import { guard } from "@/lib/guard";
import { refinePrompt } from "@/lib/refine-prompt";

export async function POST(req: Request) {
  const auth = guard.checkAuth(req);
  if (!auth.ok) return Response.json({ error: auth.error }, { status: auth.status });
  const rate = guard.checkRateLimit(req);
  if (!rate.ok) return Response.json({ error: rate.error }, { status: rate.status });

  const { draft } = await req.json();
  if (typeof draft !== "string" || draft.trim().length < 8) {
    return Response.json({ error: "draft too short" }, { status: 400 });
  }
  const refined = await refinePrompt(draft.trim());
  return Response.json({ refined });
}

  3. Wire refine state in ResearchChat/index.tsx. Track the pre-refine draft for undo; call the API; never call send() from refine:

    const [refining, setRefining] = useState(false);
const [refined, setRefined] = useState(false);
const draftBeforeRefineRef = useRef<string | null>(null);

async function onRefine() {
  const draft = input.trim();
  if (draft.length < 8 || busy || uploading || refining) return;
  draftBeforeRefineRef.current = input;
  setRefining(true);
  try {
    const res = await fetch("/api/refine-prompt", {
      method: "POST",
      headers: { "Content-Type": "application/json", ...authHeaders() },
      body: JSON.stringify({ draft }),
    });
    if (!res.ok) throw new Error("refine failed");
    const { refined: next } = await res.json();
    setInput(next);
    setRefined(true);
    textareaRef.current?.focus();
  } finally {
    setRefining(false);
  }
}

function undoRefine() {
  if (draftBeforeRefineRef.current != null) {
    setInput(draftBeforeRefineRef.current);
    draftBeforeRefineRef.current = null;
  }
  setRefined(false);
}

// Clear refined banner when user edits manually:
function onInputChange(value: string) {
  setInput(value);
  if (refined) setRefined(false);
}

  4. Add Refine + notice to ChatComposer.tsx. Button beside the paperclip; banner above the form when refined:

    {refined && (
  <div className="mb-2 flex items-center justify-between gap-2 rounded-lg bg-primary/10 px-3 py-2 text-sm text-foreground">
    <span>Refined — review before sending</span>
    <button type="button" onClick={onUndoRefine} className="text-primary underline-offset-2 hover:underline">
      Undo
    </button>
  </div>
)}
<button
  type="button"
  onClick={onRefine}
  disabled={input.trim().length < 8 || busy || uploading || refining}
  aria-label="Refine question"
  className="…"
>
  {refining ? "…" : "Refine"}
</button>

    Keep Refine as a text label or a small sparkle icon — but always with an aria-label.

  5. Test three paths.

    • Vague draft — type competitors → Refine → textarea shows a specific question → edit one word → Send runs your version
    • Undo — Refine → Undo restores the exact original draft; banner clears
    • Guardrails — Refine disabled below 8 chars, while busy, and during upload; refine never fires sendMessage on its own
What you ship

Self-check

Why not auto-send after Refine?

Enhancement is a draft, not a decision. Auto-send abdicates agency and burns compute on rewrites the user would have trimmed.

Why a separate lightweight endpoint?

Research runs tools + structured output — seconds and dollars. Refine is a single cheap text call. Mixing them couples latency and hides the UX boundary between "shape my question" and "answer it."

How is this different from lesson 7's citation post-processing?

Same philosophy: don't bet the UX on model discipline alone. L7 fixed citations deterministically after generation; L8 fixes intent explicitly before generation. Both keep the user in control.
