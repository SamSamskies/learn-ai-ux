Designing the Interrupted State

Your research assistant lets users hit Stop — then throws away what they were watching. This lesson closes the last real hole in your response lifecycle, in your actual repo.

Where you actually are

I read ResearchChat.tsx. You handle idle, submitted, thinking, streaming, complete, and error well — the thinking/streaming states even stream labeled steps ("Searching the web…") and progressive structured output, which is the advanced move. So we skip all of that. One state is undesigned: stopped.

The bug, precisely

Your rich per-turn UI only renders while inFlight = isLast && busy. When the user hits Stop, busy flips to false, so inFlight is false — but the turn's phase is still searching or streaming-answer, not done/error. Your default branch only renders done and error, so the turn collapses to just the question bubble. The partial brief the user was watching stream simply vanishes.

The principle

An interruption is a user action, not a failure. Three rules make it feel like control instead of a crash:

PreserveKeep whatever partial output already streamed. Never blank the screen on stop.
MarkSay plainly that the user stopped it — a "Stopped" badge, not an error.
RecoverOffer one clear next action. regenerate() retries the turn in place — no duplicate question.
Primary source (skim the Stop & Regenerate sections) AI SDK UI · Chatbot — Cancellation (stop) & regenerate. Note: regenerate({ messageId }) re-runs a specific assistant message and drops messages after it — perfect for retry-in-place.

Build it (~15 min, in components/ResearchChat.tsx)

  1. Pull regenerate out of useChat. You already destructure everything else.

    const { messages, sendMessage, status, stop, error, regenerate } =
  useChat<ResearchUIMessage>({ id: sessionId, transport });

  2. Add a StoppedNotice component next to LiveTurn. It reuses your existing BriefPreviewPanel and RetryIcon — preserve, mark, recover in one small piece.

    function StoppedNotice({
  state,
  onRegenerate,
}: {
  state?: ResearchUIState;
  onRegenerate: () => void;
}) {
  return (
    <div className="space-y-3">
      <div className="flex items-center gap-2">
        <span className="inline-flex items-center rounded-lg border border-outline-variant bg-surface-container-low px-2.5 py-1 text-xs font-semibold uppercase tracking-wide text-on-surface-variant">
          Stopped
        </span>
        <span className="text-sm text-on-surface-variant">
          You stopped this response.
        </span>
      </div>

      {/* Preserve: keep whatever partial output streamed */}
      {state?.briefPreview && <BriefPreviewPanel preview={state.briefPreview} />}

      {/* Recover: retry the SAME turn in place */}
      <button
        type="button"
        onClick={onRegenerate}
        className="inline-flex items-center gap-2 rounded-lg border border-outline-variant px-3 py-2 text-sm font-medium text-foreground transition-colors hover:bg-surface-container-low"
      >
        <RetryIcon />
        Regenerate
      </button>
    </div>
  );
}

  3. Detect the interrupted turn inside your turns.map(...), where you already compute research and assistant. An interrupted turn is one that has an assistant message but never reached a terminal phase:

    const interrupted =
  !!assistant &&
  research?.phase !== "done" &&
  research?.phase !== "error";

    Because the in-flight turn is already returned earlier via LiveTurn, anything reaching this point is genuinely stopped — even an older turn you stopped before asking something new.

  4. Render it in the default branch, alongside your existing done and error blocks:

    {interrupted && (
  <StoppedNotice
    state={research}
    onRegenerate={() => regenerate({ messageId: assistant!.id })}
  />
)}

  5. Test the loop. Ask something, hit Stop mid-search and mid-stream. You should see the question, the partial brief preview (if any), a "Stopped" badge, and Regenerate — which replaces the turn in place with a fresh answer. No duplicate question.

Bonus fix (2 min) — kills gap #2

Your error retry currently calls retryQuestion(question), which send()s a new user turn (duplicate question, stale banner left behind). Swap the domain-error ErrorBanner's onRetry to () => regenerate({ messageId: assistant.id }) for the same in-place behavior you just built.

What we're NOT doing (and why)

True Continue — resuming generation from the exact partial — needs server support (resumable streams / a "continue from here" endpoint). Your brief is a single structured generation, so Regenerate is the honest recovery action here. If you later want real resume, that's the AI SDK's resumable-streams territory — a future lesson.

Self-check

After Stop, why does the partial brief disappear in the current code — name the exact condition.

inFlight = isLast && busy becomes false on stop, but phase is still searching/streaming-answer. The default render branch only matches phase === "done" or "error", so nothing but the user bubble renders.

Why use regenerate with a messageId instead of re-sending the question text?

Re-sending appends a new user turn (duplicate question + stale turn left behind). regenerate({ messageId }) drops messages after that assistant message and retries that turn in place — one question, one answer.
