AI Product UX · Phase 2 · Lesson 0004

Legible Thinking Gap

Every lifecycle state is deliberate now. The next win is perceived speed — splitting the wait into three honest micro-states so users never stare at silence or a generic spinner.

Where you actually are

Your research assistant already streams labeled steps (searching documents, searching the web) and skeleton cards — better than most demos. But two gaps remain in the early wait:

Gap 1 — submitted looks like working

When status === 'submitted' and no assistant message exists yet, you render SearchingStatus with "Working…". That promises backend progress you don't have. The user sent a question; the HTTP round-trip hasn't even opened.

Gap 2 — silent thinking gap

Your turn loop only renders a live indicator when research exists. But the SDK creates an assistant message before the first data-research part arrives. In that window the user sees their question and… nothing. This is exactly the thinking gap from lesson 1.

The principle

Perceived latency is how slow it feels. You can't always make the model faster, but you can always make the wait legible. Three rules:

Acknowledge instantlyEcho the user's message and disable resend on Send — you already do this.
Name the wait honestlySubmitted → lightweight ack. Thinking → distinct indicator. Working → named steps once you know them.
Keep context in viewAuto-scroll to the latest turn on send so the user sees their question and whatever indicator follows.

One spinner for all three micro-states feels slower than three honest treatments — even when wall-clock time is identical.

Primary source (skim both) AI SDK · Streaming status delay — why status flips to streaming before content arrives. Pair with The Trust-Latency Gap for the framing: speed without legibility erodes trust.

Build it (~15 min)

  1. Create lib/derive-in-flight-phase.ts. One function that owns "what kind of wait is this?" — SDK status + optional research snapshot.

    import type { ChatStatus } from "ai";
import type { ResearchUIState } from "@/lib/research-state";

export type InFlightPhase = "submitted" | "thinking" | "working";

export function deriveInFlightPhase(
  status: ChatStatus,
  research?: ResearchUIState,
): InFlightPhase {
  if (status === "submitted") return "submitted";

  // Stream open — but no app progress yet
  if (!research) return "thinking";
  if (
    research.phase === "searching" &&
    !research.searched &&
    !research.searchedDocs
  ) {
    return "thinking";
  }

  return "working";
}

  2. Add two small indicators in ResearchChat/index.tsx — keep them visually lighter than SearchingStatus:

    function SubmittedAck() {
  return (
    <div className="flex items-center gap-2 text-sm text-on-surface-variant">
      <Spinner />
      <span>Got it — starting research…</span>
    </div>
  );
}

function ThinkingGap() {
  return (
    <div className="flex items-center gap-2 text-sm text-on-surface-variant">
      <Spinner />
      <span>Thinking…</span>
    </div>
  );
}

    No skeleton card here — you don't know the output shape yet. Save the skeleton for working when search is actually underway.

  3. Refactor the in-flight turn render — one branch for all busy turns, keyed on deriveInFlightPhase:

    if (inFlight) {
  const waitPhase = deriveInFlightPhase(status, research);

  return (
    <div key={turn.user.id} className="flex flex-col gap-6">
      <UserMessage text={question} />

      {waitPhase === "submitted" && <SubmittedAck />}
      {waitPhase === "thinking" && <ThinkingGap />}
      {waitPhase === "working" && research && (
        <SearchingStatus
          searched={research.searched}
          searchedDocs={research.searchedDocs}
        />
      )}
      {waitPhase === "working" && research?.phase === "streaming-answer" && (
        research.briefPreview && (
          <BriefPreviewPanel preview={research.briefPreview} defaultOpen />
        )
      )}
    </div>
  );
}

    This replaces the separate inFlight && research / inFlight && !assistant branches and closes the silent gap.

  4. Auto-scroll on send. A ref on the scroll container + scrollIntoView on the latest turn anchor:

    const bottomRef = useRef<HTMLDivElement>(null);

function send(text: string) {
  // …existing send logic…
  requestAnimationFrame(() => {
    bottomRef.current?.scrollIntoView({ behavior: "smooth", block: "end" });
  });
}

// inside the messages column, after the turns map:
<div ref={bottomRef} aria-hidden="true" />

  5. Test the three micro-states. Throttle Network → Slow 3G, send a question, and watch the sequence:

    • Submitted: "Got it — starting research…" (brief, no skeleton)
    • Thinking: "Thinking…" (stream open, no search flags yet)
    • Working: "Searching documents…" / skeleton card (your existing labeled steps)

    Your question should stay in view the whole time.

What you shipped

Honest progress during the wait — the first lever of perceived latency. Next up: trust signals (citations, source surfacing, calibrated uncertainty) or human-in-the-loop flows, depending on what your retrofit needs most.

Self-check

Why is "Working…" during submitted misleading, even though something is technically happening?

It implies named backend progress (search, retrieval) before you have any signal of that. Users interpret it as a stall when the label doesn't change. A lightweight ack matches what you actually know.

When does the silent thinking gap occur in your app, and what signal fixes it?

After the SDK creates an assistant message but before the first data-research part arrives — your old branch required research to exist. deriveInFlightPhase returns thinking when research is undefined during streaming.
