Latency Budget & VAD Tuning

Lesson 0004 taught you to name the thinking gap on text. Voice adds a fixed cost before thinking even starts: VAD silence detection. Today's win: a turn-latency readout on your probe plus one tuned silence_duration_ms you can defend with numbers.

Why this, why now

Barge-in, partial transcripts, phase badges — the probe looks like a product. But if replies feel sluggish, users blame the model when the culprit is often 700 ms of configured silence sitting on top of the thinking gap.

  • You can't optimize latency you don't measure.
  • silence_duration_ms is a product knob — shorter feels snappier, longer respects mid-sentence pauses.
  • threshold is an environment knob — noisy rooms need a higher gate.

Same perceived-latency playbook as text. Different stack to budget.

The core idea

Turn latency = VAD tail + model gap. Stamp speech_stopped, stop the clock on first audio. Tune VAD against a ~500–800 ms conversational budget — not vibes.

Primary source — read this (6 min) OpenAI · Voice activity detection (VAD) — focus on server_vad knobs and the tradeoff note on silence_duration_ms. Cheat sheet: latency budget reference.

Know this first (~5 min)

What you're measuring

EventMeaning
input_audio_buffer.speech_stopped Server decided you finished talking — start the clock.
response.output_audio.delta (or transcript delta) Model is responding — stop the clock.

This span includes VAD tail remainder + model reasoning + network. It's the number users feel as "how fast is this thing?"

The VAD tradeoff

SymptomDialDirection
Feels sluggish after I stop talking silence_duration_ms Lower (try 450–500)
Model jumps in while I'm pausing mid-thought silence_duration_ms Raise (try 700–900)
False barge-in / background noise triggers speech threshold Raise (you're already at 0.75 — good for desktop)

Your session route already sets silence_duration_ms: 700. That's patient — fine for thoughtful answers, heavy for quick back-and-forth.

Build it (~20 min)

Two changes: a latency readout in VoiceProbe, then one VAD tweak you validate with before/after numbers.

  1. Add a turn-latency timer. In VoiceProbe/index.tsx, track when the user finishes and when audio starts:

    const speechStoppedAt = useRef<number | null>(null);
const [turnLatencyMs, setTurnLatencyMs] = useState<number | null>(null);

function trackLatency(event: { type: string }) {
  if (event.type === 'input_audio_buffer.speech_stopped') {
    speechStoppedAt.current = performance.now();
    return;
  }
  const isFirstAudio =
    event.type === 'response.output_audio.delta' ||
    event.type === 'response.audio.delta' ||
    event.type === 'output_audio_buffer.started';
  if (isFirstAudio && speechStoppedAt.current != null) {
    setTurnLatencyMs(Math.round(performance.now() - speechStoppedAt.current));
    speechStoppedAt.current = null;
  }
}

    Call trackLatency(event) inside dc.onmessage before your reducers. Reset speechStoppedAt on disconnect.

  2. Render the readout under the phase badge. Muted helper text — a debug surface today, a perf regression guard tomorrow:

    {turnLatencyMs != null && (
  <p className="mt-3 text-center text-sm text-on-surface-variant">
    Last turn: {turnLatencyMs} ms
    {turnLatencyMs > 800 && (
      <span className="text-warn"> — over budget</span>
    )}
  </p>
)}

    800 ms is a rule of thumb, not a SLA. Use it to spot regressions when you change config.

  3. Tune silence_duration_ms and re-measure. In app/api/realtime/session/route.ts, drop from 700 to 500 (OpenAI default):

    silence_duration_ms: 500,

    Reconnect, ask three short questions, note the readout each time. Expect ~200 ms improvement — that's pure VAD tail, not magic model speed.

    If the model now cuts you off on deliberate pauses, split the difference (550–600) or bump back up. Pick the value you can explain: "500 ms silence — snappy Q&A; I'll raise it if users pause mid-thought."

  4. Smoke-test the tradeoffs.

    • Short question: "What's two plus two?" — latency readout under ~800 ms feels conversational.
    • Mid-sentence pause: "The capital of France is… Paris." — if the model responds at "is", raise silence_duration_ms.
    • Noisy room (optional): if background chatter triggers false listening, raise threshold slightly instead of touching silence first.
Watch out

Timer never fires? Confirm you're listening for the exact event types your model emits (response.audio.delta vs response.output_audio.delta). Readout stuck high after VAD change? Most of the gap may be model thinking — VAD only shaves the configured silence. Comparing presets? Disconnect and reconnect — session config is set at mint.

Self-check

Why measure speech_stopped → first audio, not speech_started → first audio?

Users care about response time after they finish talking. Including their speech duration would blame them for a slow model.

Does a lower silence_duration_ms always feel better?

No — it trades snappiness for premature turn-taking. List-heavy or reflective use cases often want 700+ ms; quick command-and-response wants 400–500.

When would you switch to semantic_vad?

When fixed silence can't distinguish "I'm thinking" pauses from "I'm done" — e.g. open-ended storytelling. Start with server_vad until that's clearly broken.
