Voice Image Output

Lesson 0017 attached images into the session. Today the model asks to generate one out. Win: a gated generate_illustration tool, editable prompt on screen (because STT will mangle it), then the picture in the transcript while voice narrates.

Why this, why now

Mission multimodal I/O is half-done: input shipped, output hasn't. Realtime already speaks — that's audio out. What's missing is visual companion media: the screen shows what speech can't carry (a diagram, a sketch, a chart mock).

  • Images don't come from Realtime audio — your client runs a tool that calls the Images API, then you render the result.
  • Spoken prompts are lossy. Confirming the prompt on screen is the product fix, not hoping transcription improves.
  • Pattern: voice for the headline, screen for the picture — same rule as lists and URLs.
The core idea

Multimodal output in voice is propose prompt → edit on screen → generate → show + speak. Never fire image gen from an unverified transcript.

Primary source — read this section (6 min) OpenAI · Image generation — skim Generate Images + customize output (quality, size). Pair with AI Design Patterns · Voice ("When Voice Needs a Visual Companion"). Cheat sheet: multimodal voice output reference.

Know this first (~4 min)

Same tool loop, longer pause

Reuse lessons 0018–0019: detect function_call → hold for approval → execute → function_call_output → response.create. Image gen can take several seconds — dock label should say Generating image…, not "Looking up…".

Why the prompt must be editable

Staging a research query is short and checkable. An illustration prompt is long and creative — one misheard word ruins it and costs tokens. Show a textarea prefilled with the model's prompt arg. Approve sends your edited string to the API.

Keep generation cheap for the prototype

Use gpt-image-1-mini, quality: "low", size: "1024x1024". Return b64_json and display as a data: URL. One image per call — Shape of AI's Variations pattern is out of scope for this lesson.

Build it (~20 min)

  1. Add the tool + helpers. Extend lib/realtime-voice-tools.ts:

    export const GENERATE_ILLUSTRATION_TOOL = {
  type: "function" as const,
  name: "generate_illustration",
  description:
    "Create a simple illustration or diagram when the user asks to draw, sketch, visualize, or generate an image. Pass a detailed visual prompt.",
  parameters: {
    type: "object",
    properties: {
      prompt: {
        type: "string",
        description: "Detailed description of the image to generate",
      },
    },
    required: ["prompt"],
  },
};

export const VOICE_TOOLS = [
  LOOKUP_DEFINITION_TOOL,
  STAGE_RESEARCH_BRIEF_TOOL,
  GENERATE_ILLUSTRATION_TOOL,
] as const;

export function toolApprovalLabel(name: string): string {
  if (name === STAGE_RESEARCH_BRIEF_TOOL.name) {
    return "Waiting for your approval…";
  }
  if (name === GENERATE_ILLUSTRATION_TOOL.name) {
    return "Confirm the image prompt…";
  }
  return "Waiting for approval…";
}

export function generatingImageLabel(): string {
  return "Generating image…";
}

  2. Server route for Images API. Create app/api/voice/generate-image/route.ts (keep the key server-side):

    import OpenAI from "openai";
import { NextResponse } from "next/server";

const openai = new OpenAI();

export async function POST(req: Request) {
  const { prompt } = (await req.json()) as { prompt?: string };
  const trimmed = (prompt ?? "").trim();
  if (!trimmed) {
    return NextResponse.json({ error: "prompt required" }, { status: 400 });
  }

  const result = await openai.images.generate({
    model: "gpt-image-1-mini",
    prompt: trimmed,
    size: "1024x1024",
    quality: "low",
  });

  const b64 = result.data?.[0]?.b64_json;
  if (!b64) {
    return NextResponse.json({ error: "no image" }, { status: 502 });
  }

  return NextResponse.json({
    imageDataUrl: `data:image/png;base64,${b64}`,
    prompt: trimmed,
  });
}

  3. Transcript can hold an image. Extend Turn and add a helper:

    export type Turn = {
  role: "user" | "assistant" | "system";
  text: string;
  interrupted?: boolean;
  imageUrl?: string;
};

export function appendImageTurn(
  state: TranscriptState,
  prompt: string,
  imageUrl: string,
): TranscriptState {
  return {
    ...state,
    history: [
      ...state.history,
      { role: "assistant", text: prompt, imageUrl },
    ],
  };
}

    In Transcript.tsx, when turn.imageUrl is set, render an <img> under the text (alt = prompt). Keep layout simple — full width of the assistant column, rounded corners matching existing chrome.

  4. Approval UI with editable prompt. Don't reuse plain ConfirmDialog for this — you need a textarea. Add a small ImageGenApproval.tsx (or inline in VoiceProbe): title "Generate this image?", controlled textarea seeded from the tool arg, Cancel / Generate buttons. On open, dock label = toolApprovalLabel(GENERATE_ILLUSTRATION_TOOL.name).

  5. Wire the branch in VoiceProbe. Extend pending state (or a second piece of state) for image gen:

    type PendingImageGen = {
  call: RealtimeFunctionCall;
  prompt: string;
};

// in handleFunctionCall:
if (call.name === GENERATE_ILLUSTRATION_TOOL.name) {
  const prompt = String(call.args.prompt ?? "").trim();
  setPendingImage({ call, prompt });
  setToolLabel(toolApprovalLabel(call.name));
  return;
}

async function resolveImageGen(approved: boolean, editedPrompt: string) {
  const pending = pendingImage;
  if (!pending) return;
  setPendingImage(null);

  const dc = dcRef.current;
  if (dc?.readyState !== "open") return;

  if (!approved) {
    setToolLabel(null);
    dc.send(JSON.stringify(rejectedToolOutput(pending.call.callId)));
    dc.send(JSON.stringify({ type: "response.create" }));
    return;
  }

  setToolLabel(generatingImageLabel());
  try {
    const res = await fetch("/api/voice/generate-image", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ prompt: editedPrompt }),
    });
    const data = await res.json();
    if (!res.ok || !data.imageDataUrl) throw new Error(data.error ?? "failed");

    setTranscript((t) => appendImageTurn(t, editedPrompt, data.imageDataUrl));
    dc.send(JSON.stringify(
      functionOutputEvent(pending.call.callId, {
        ok: true,
        note: "Image is on screen. Describe it in one short spoken sentence.",
      }),
    ));
  } catch {
    dc.send(JSON.stringify(
      functionOutputEvent(pending.call.callId, {
        ok: false,
        error: "generation_failed",
      }),
    ));
  }
  setToolLabel(null);
  dc.send(JSON.stringify({ type: "response.create" }));
}

    Register the tool in session instructions: call generate_illustration when the user asks to draw/sketch/visualize; if rejected, acknowledge and stay in voice.

  6. Smoke test.

    • Say "Draw a simple diagram of how RAG works" — approval opens with a prompt; edit one word before Generate.
    • Dock shows Confirm → Generating… → image appears in transcript; assistant speaks a one-liner.
    • Cancel path: no image, spoken acknowledgment, session still live.
    • Lookup and stage_research_brief still behave as before.
Watch out

Don't auto-run image gen from raw STT — that's how you burn money on gibberish prompts. Don't put the API key in client code. Clear pending image approval on disconnect. If generation fails, still complete the function-call loop so the model can apologize aloud.

Self-check

Why not generate inside Realtime itself?

Realtime is speech-to-speech. Image bytes come from the Images API (or a Responses image tool). Your client owns the tool run and the UI that shows the result.

Why edit the prompt on screen?

Voice transcription is lossy for creative detail. The editable prompt is strategic friction that matches cost × irreversibility of a bad generation.

What does the user see vs hear?

Screen: the image (and prompt caption). Voice: a short spoken description — headline only.
