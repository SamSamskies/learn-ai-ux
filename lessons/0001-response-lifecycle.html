Designing the Response Lifecycle

Your model already streams. This lesson turns that raw mechanic into a deliberate, five-state experience — the foundation the rest of your retrofit builds on.

Why this, why first

You've already shipped streaming in Phase 1 — a streaming UI state machine, a research assistant that streams, an AI SDK server. So we're not teaching streaming. We're teaching the design frame that separates a demo from a product: an AI reply is not a request-and-response, it's a lifecycle that passes through distinct states, and each state is a UI decision you make on purpose.

Most AI UIs are competent at two states (empty and full text) and accidental about the rest. Trust leaks out through the gaps: the silent spinner, the raw error, the reply you can't stop. Closing those gaps is the single highest-leverage thing your frontend background lets you do better than a backend engineer.

The core idea

A nondeterministic reply moves through idle → submitted → thinking → streaming → complete, with stopped and error as exits. Ship a treatment for every box, not just the pretty two.

The state most people miss

Here's the non-obvious one. The Vercel AI SDK flips status to 'streaming' the instant the connection opens — which includes metadata, before a single token of text arrives. If you render your "typing" UI on status === 'streaming', users stare at an empty, misleading state during what we'll call the thinking gap — and that silence is exactly where confidence collapses.

The fix: distinguish thinking from typing by looking at whether the last assistant message actually has content yet.

Primary source — read this (5 min) AI SDK UI · Chatbot & the status property — plus the short "streaming status but no text" note that explains the thinking gap.

Hands-on: give your research assistant all five states (~15 min)

We'll retrofit the reply surface of your Phase 1 research assistant so every state is intentional. This is a self-contained component you can drop in and adapt.

  1. Derive the real state. Don't trust status alone — combine it with message content to split thinking from streaming.

    // useLifecycle.ts — one place that owns "what state are we really in?"
import { useChat } from '@ai-sdk/react';

export type Phase =
  'idle' | 'submitted' | 'thinking' | 'streaming' | 'error';

export function derivePhase(chat: ReturnType<typeof useChat>): Phase {
  const { status, messages } = chat;
  if (status === 'error') return 'error';
  if (status === 'submitted') return 'submitted';
  if (status === 'streaming') {
    const last = messages.at(-1);
    const hasText = last?.role === 'assistant' &&
      last.parts?.some((p) => p.type === 'text' && p.text.length > 0);
    return hasText ? 'streaming' : 'thinking'; // ← the gap
  }
  return 'idle';
}

  2. Render a treatment for each state. Notice the composer, indicator, and actions all change per state — that's the whole point.

    const chat = useChat();
const phase = derivePhase(chat);
const { messages, error, sendMessage, stop, regenerate } = chat;

return (
  <div>
    {messages.map((m) => <Message key={m.id} message={m} />)}

    {phase === 'submitted' && <Acknowledge />}      {/* echoed instantly */}
    {phase === 'thinking' && <ThinkingDots label="Thinking…" />}
    {phase === 'streaming' && <StopButton onClick={stop} />}
    {phase === 'error' && (
      <ErrorCard
        message={friendly(error)}                 {/* never a raw stack */}
        onRetry={() => regenerate()}                {/* input preserved */}
      />
    )}

    <Composer
      disabled={phase === 'submitted' || phase === 'thinking'}
      onSend={(text) => sendMessage({ text })}
    />
  </div>
);

  3. Make the thinking gap honest. If your backend does real work before the first token (retrieval, tool calls), stream a step label instead of generic dots — "Searching sources…", "Reading 4 results…". Legible progress beats a spinner. (We go deep on this in a later lesson; for now, a labelled indicator is the win.)

  4. Design the stop. After stop(), keep the partial text on screen, tag it as stopped, and offer Continue / Regenerate. A reply the user can't interrupt feels like a runaway process, not a tool.

Watch out

If status is stuck on 'ready' and never changes, you likely mutated the useChat id right before sending. Keep the id stable across a send. (Known footgun — see the AI SDK issue tracker.)

Self-check

Your UI shows the typing cursor, but no text appears for three seconds. Which state are you actually in, and what went wrong?

You're in the thinking state, not streaming. The SDK set status to 'streaming' on connection open (metadata), before tokens arrived. You rendered typing UI on status alone instead of also checking that the last assistant message has non-empty text parts.

Name the two lifecycle exits that most demos forget, and the one rule that applies to both.

stopped and error. The shared rule: preserve the user's work — keep partial output on stop; keep the input and offer retry on error. Never lose what the user gave you.
