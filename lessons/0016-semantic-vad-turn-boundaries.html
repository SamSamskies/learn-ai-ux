Semantic VAD & Turn Boundaries

You tuned silence_duration_ms to 500 ms for snappy Q&A. That same knob makes the model jump in when someone pauses mid-thought. Today's win: switch to semantic_vad, pick an eagerness that fits research-style speech, and surface the mode so users know why pauses behave differently.

Why this, why now

Lesson 0013 gave you a latency readout and a defended silence value. Lesson 0015 closed the mode-switch arc. The remaining voice friction is often turn boundaries — when the session decides you're done talking.

  • server_vad adds a fixed silence tail — great for commands, brittle for reflective answers.
  • Raising silence fixes mid-thought cutoffs but makes every short question feel sluggish.
  • semantic_vad trades predictable timing for smarter boundaries — the right tool for exploratory research Q&A.
The core idea

Turn detection is a product knob, not a one-time default. Pick the mode that matches how users actually speak in this surface — and make the choice legible in the UI.

Primary source — read this (5 min) OpenAI · Voice activity detection (VAD) — focus on the Semantic VAD section and the eagerness table. Cheat sheet: semantic VAD reference.

Know this first (~4 min)

What semantic VAD does differently

Instead of counting milliseconds of silence, the API scores whether your words sound like a complete utterance. A definitive statement gets a short wait; a trailing "umm…" gets a longer one. Same events — speech_started / speech_stopped — but different timing logic behind them.

The tradeoff you accept

server_vadsemantic_vad
Latency predictability Fixed tail ≈ silence_duration_ms Content-dependent — readout will vary turn-to-turn
Mid-thought pauses Often cuts off unless silence is high Usually waits through "…is" before "Paris"
Tuning dial silence_duration_ms, threshold eagerness (low / medium / high)

For a research assistant voice mode — open questions, thinking aloud — start at eagerness: 'low'.

Build it (~20 min)

  1. Centralize turn-detection config. Add lib/realtime-turn-detection.ts so session mint and UI share one source of truth:

    export const REALTIME_TURN_DETECTION = {
  type: 'semantic_vad' as const,
  eagerness: 'low' as const,
  interrupt_response: true,
};

export function turnDetectionLabel(): string {
  const { type, eagerness } = REALTIME_TURN_DETECTION;
  if (type === 'semantic_vad') {
    return `Turn detection: semantic (${eagerness} eagerness)`;
  }
  return 'Turn detection: server VAD';
}

  2. Swap VAD mode in the session route. In app/api/realtime/session/route.ts, import the constant and replace the server_vad block:

    import { REALTIME_TURN_DETECTION } from '@/lib/realtime-turn-detection';

// inside audio.input:
turn_detection: REALTIME_TURN_DETECTION,

    Remove threshold, prefix_padding_ms, and silence_duration_ms — they apply only to server_vad. Disconnect and reconnect to pick up the new session.

  3. Surface the mode in VoiceProbe. Import turnDetectionLabel and render it under your latency readout — muted helper text, same tier as "Last turn: 612 ms":

    <p className="mt-2 text-center text-xs text-on-surface-variant">
  {turnDetectionLabel()}
</p>

    This is a trust signal for voice: users learn why a pause didn't trigger a reply yet, or why latency isn't fixed, without seeing raw config.

  4. Run the two smoke tests from lesson 0013 — compare outcomes.

    • Mid-sentence pause: "The capital of France is… Paris." — with server_vad at 500 ms this often triggered at "is". Semantic + low eagerness should wait for "Paris".
    • Short factual: "What's two plus two?" — should still feel conversational. If sluggish, try eagerness: 'medium' and reconnect.
    • Trailing filler (bonus): "I think the answer is umm… four." — semantic VAD should tolerate the filler better than fixed silence.

    Note latency numbers in a scratch comment or your own notes — expect more variance than with server_vad. That's the tradeoff, not a regression.

Watch out

Don't flip modes without reconnecting — session config is set at mint. Don't chase identical latency readouts on semantic_vad; judge by cut-off behavior and conversational feel. If both modes feel wrong, check mic hygiene (echo cancellation, noise reduction) before tuning further.

Self-check

Why not just raise silence_duration_ms to 900 ms?

It fixes mid-thought pauses but adds ~400 ms to every short question — the snappy Q&A you validated in lesson 0013. Semantic VAD targets the pause problem without penalizing every turn.

When would you switch back to server_vad?

Command-and-response surfaces where predictable sub-500 ms tails matter more than reflective speech — e.g. a voice calculator or fast tool invocations.

What does surfacing turn detection buy the product?

Legibility. Users can't see session JSON — a label explains behavior changes after config updates and reduces "is it broken?" reports when pauses feel longer.
