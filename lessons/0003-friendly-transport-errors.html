Friendly Transport Errors

You designed domain errors and the interrupted state. One lifecycle exit still leaks raw SDK text — the transport failure path. This lesson closes it.

Where you actually are

After lesson 2, every terminal state in your turn loop is deliberate: done, domain error, and stopped. Domain errors already show a friendly fallback string and retry with regenerate({ messageId }).

But the bottom banner — the one driven by useChat's top-level error — still does two wrong things:

The gap, precisely

Raw message: message={error.message} can surface Failed to fetch, HTTP status codes, or JSON parse errors — technical text that reads like a bug, not a product.

Wrong retry: retryQuestion(getUserText(last.user)) calls send(), appending a duplicate user turn — the exact anti-pattern you fixed everywhere else.

The principle

Your app has two error channels. Treat them differently in the UI, but the user-facing rules are the same:

Domain errorsYour server streams phase: 'error' inside data-research. You control the copy via research.error.
Transport errorsThe fetch/stream itself failed — network, auth, 5xx, SDK parse. useChat sets top-level error. You must translate SDK text into product copy.

Three rules for any error surface:

Plain languageSay what happened and what to do — never expose stack traces, status codes, or fetch internals.
Preserve contextKeep the user's question visible. Don't wipe the thread on failure.
Retry in placeregenerate({ messageId }) when an assistant message exists; only re-send when the request failed before one was created.
Primary source (skim the Error Handling section) AI SDK UI · Chatbot — Error Handling — documents the top-level error object and regenerate for retry. Pair with NN/g · Error-Message Guidelines for the copy rules (specific, blame-free, actionable).

Build it (~15 min)

  1. Create lib/friendly-transport-error.ts. One function, one fallback. Match on error.name, error.message substrings, and optional HTTP status if your transport attaches it.

    const DEFAULT =
  "Something went wrong. Please try again.";

export function friendlyTransportError(error: Error): string {
  const msg = error.message.toLowerCase();

  if (error.name === "AbortError") {
    return "The request was cancelled.";
  }
  if (
    msg.includes("failed to fetch") ||
    msg.includes("network") ||
    msg.includes("load failed")
  ) {
    return "Connection problem — check your network and try again.";
  }
  if (msg.includes("401") || msg.includes("403") || msg.includes("unauthorized")) {
    return "You're not signed in. Refresh the page and try again.";
  }
  if (msg.includes("429") || msg.includes("rate limit")) {
    return "Too many requests — wait a moment and try again.";
  }
  if (msg.includes("500") || msg.includes("502") || msg.includes("503")) {
    return "Our servers hit a snag. Try again in a few seconds.";
  }

  return DEFAULT;
}

    Log the raw error to the console in dev if you want — just never show it to users.

  2. Wire it into the transport banner in ResearchChat/index.tsx:

    import { friendlyTransportError } from "@/lib/friendly-transport-error";

{error && (
  <div className="mt-6">
    <ErrorBanner
      message={friendlyTransportError(error)}
      onRetry={retryTransportError}
    />
  </div>
)}

  3. Extract retryTransportError — regenerate in place when you can, re-send only as a last resort:

    function retryTransportError() {
  const last = turns.at(-1);
  if (!last) return;

  if (last.assistant) {
    regenerate({ messageId: last.assistant.id });
    return;
  }

  // Failed before an assistant message existed — safe to re-send once
  send(getUserText(last.user));
}

    This mirrors the pattern you already use for domain errors and stopped turns.

  4. Test both channels.

    • Domain error: trigger a search failure your server handles (you already have friendly copy + regenerate).
    • Transport error: in DevTools → Network, set throttling to Offline, send a question, hit Try again with network restored. You should see plain language, your question once, and no duplicate bubble on retry.

Self-check

Why are there two error paths in your chat, and why can't you delete one?

Domain errors are part of your streamed research state — the server completed the HTTP response but reported failure inside the brief pipeline. Transport errors mean the HTTP/stream layer itself failed before or during delivery. Different origins, same user-facing rules.

When is re-sending the question text the correct retry, and when is it wrong?

Wrong when an assistant message already exists — that duplicates the user turn. Correct only when the request failed before any assistant message was created, so regenerate({ messageId }) has nothing to target.
