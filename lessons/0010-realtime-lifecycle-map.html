Realtime Lifecycle Map

Text taught you seven deliberate states. Voice adds listening, speaking, and barge-in — same trust problem, harder latency budget. This lesson maps the lifecycle and ships a minimal voice probe that makes every phase visible.

Why this, why now

You closed the text arc with a polished research assistant: lifecycle states, trust signals, human-in-the-loop. Voice is the same design problem — users still need to know where they are in a nondeterministic wait — but audio raises the stakes. Silence isn't ambiguous; it feels broken. A 700 ms gap without feedback kills conversational trust faster than a spinner on text.

Before you wire a full voice agent into your app, you need a probe: a tiny surface whose only job is to connect, listen, and show phases honestly. That's today's win.

The core idea

Derive phase from Realtime events on the WebRTC data channel — the voice equivalent of derivePhase on useChat. Render a treatment for every phase, not one mic button and hope.

Primary source — read this (10 min) OpenAI · Voice agents — skim the architecture table (speech-to-speech vs chained), then the browser WebRTC flow. See also the realtime lifecycle reference.

Know this first (~5 min)

Two paths to voice

ArchitecturePick when…
Speech-to-speech (Realtime API + WebRTC) You want natural turn-taking, barge-in, and lowest latency. This arc.
Chained (STT → your text agent → TTS) You need approval gates, durable transcripts, or you're extending the research chat as-is.

Text states → voice states

Your response lifecycle still applies. Voice adds connecting (WebRTC handshake), listening (user speech detected), speaking (model audio out), and interrupted (barge-in — partial transcript preserved, not an error). Full mapping: realtime lifecycle reference.

Build it (~20 min)

Add a isolated /voice route to your research assistant — not merged into the chat yet. One focused diff: server mints an ephemeral session; client connects via WebRTC and shows phase from events.

  1. Mint ephemeral credentials server-side. Never put your API key in the browser. Add app/api/realtime/session/route.ts:

    import { NextResponse } from 'next/server';

export async function POST() {
  const res = await fetch('https://api.openai.com/v1/realtime/client_secrets', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${process.env.OPENAI_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      session: {
        type: 'realtime',
        model: 'gpt-realtime-2.1',
        audio: { output: { voice: 'marin' } },
      },
    }),
  });
  if (!res.ok) return NextResponse.json({ error: 'mint failed' }, { status: 500 });
  return NextResponse.json(await res.json());
}

  2. Derive phase from data-channel events. Create lib/realtime-phase.ts — one reducer, same pattern as text derivePhase:

    export type RealtimePhase =
  'idle' | 'connecting' | 'listening' | 'thinking'
  | 'speaking' | 'interrupted' | 'error';

export function reduceRealtimePhase(
  prev: RealtimePhase,
  event: { type: string },
): RealtimePhase {
  switch (event.type) {
    case 'session.created': return 'idle';
    case 'input_audio_buffer.speech_started': return 'listening';
    case 'input_audio_buffer.speech_stopped': return 'thinking';
    case 'response.output_audio.delta':
    case 'response.output_audio_transcript.delta':
      return 'speaking';
    case 'response.done': return 'idle';
    case 'response.cancelled': return 'interrupted';
    case 'error': return 'error';
    default: return prev;
  }
}

  3. Connect with WebRTC + show the phase. Add components/VoiceProbe.tsx and a page at app/voice/page.tsx. Core connect flow (ephemeral token → peer connection → data channel):

    const [phase, setPhase] = useState<RealtimePhase>('idle');
const pcRef = useRef<RTCPeerConnection | null>(null);

async function connect() {
  setPhase('connecting');
  const { value: token } = await fetch('/api/realtime/session', { method: 'POST' }).then(r => r.json());

  const pc = new RTCPeerConnection();
  pcRef.current = pc;
  const audio = new Audio(); audio.autoplay = true;
  pc.ontrack = (e) => { audio.srcObject = e.streams[0]; };

  const mic = await navigator.mediaDevices.getUserMedia({ audio: true });
  pc.addTrack(mic.getTracks()[0]);

  const dc = pc.createDataChannel('oai-events');
  dc.onmessage = (e) => {
    const event = JSON.parse(e.data);
    setPhase((p) => reduceRealtimePhase(p, event));
  };

  const offer = await pc.createOffer();
  await pc.setLocalDescription(offer);
  const sdp = await fetch('https://api.openai.com/v1/realtime/calls', {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/sdp' },
    body: offer.sdp,
  }).then(r => r.text());
  await pc.setRemoteDescription({ type: 'answer', sdp });
}

    Render a large phase badge — reuse your thinking-dots / status-chip patterns from text. Each phase gets distinct copy: "Connecting…", "Listening…", "Thinking…", "Speaking…", "Interrupted — keep talking".

  4. Smoke-test the lifecycle. Run npm run dev, open /voice, connect, speak a short question. Watch the badge move: connecting → listening → thinking → speaking → idle. Talk over the model mid-reply — confirm interrupted appears (server-side VAD handles barge-in; your job is to show it).

Watch out

Mic permission denied → map to error with "Microphone access needed" — never a raw DOMException. Connection drops → offer Reconnect, don't leave phase stuck on speaking.

Self-check

Why derive phase from events instead of guessing from audio volume?

Client-side volume is noisy (background sound, device differences). The Realtime API emits authoritative turn events — same reason you don't trust status === 'streaming' alone for the text thinking gap.

Speech-to-speech vs chained — when would you keep your research chat text-only and chain STT/TTS around it?

When you need edit-before-send, inline citations, or tool approval on the text before anything is spoken — the research assistant's trust surfaces are text-native. Chained keeps each stage visible.

What's the voice equivalent of the text thinking gap?

The interval between speech_stopped and the first output_audio.delta — user has finished talking but hears nothing. Show "Thinking…"; budget ~500–800 ms before it feels broken.
