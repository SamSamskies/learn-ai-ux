AI Product UX · Phase 2 · Lesson 0014

Voice in the Main App

Your probe has lifecycle, transcripts, barge-in, and a tuned 500 ms VAD. Isolated /voice still feels like a lab. Today's win: one product chrome, Research | Voice modes, and session instructions that make spoken answers fit a research assistant.

Why this, why now

Merging Realtime into the text research pipeline would throw away what you already built well — citations, edit-before-send, destructive confirms, structured briefs. Those are text-native trust surfaces.

  • Speech-to-speech wins on latency and barge-in; it loses on mid-turn HITL and source lists.
  • Chained STT→agent→TTS reuses the text agent but adds latency you just spent four lessons shaving.
  • The product move for v1: sibling modes, same brand — not one Frankenstein transport.

Same retrofit mindset as the text arc. Different surface, shared chrome.

The core idea

Productize the probe with a mode switch, not a pipeline merge. Voice = exploratory Q&A. Research = briefs and sources. Instruct the voice agent to stay short and hand depth back to text.

Primary source — read this (6 min) OpenAI · Voice agents — focus on the speech-to-speech vs chained table and when each wins. Companion UX: voice for the headline, screen for the data. Cheat sheet: voice mode switch.

Know this first (~5 min)

Two transports, one product

ModeTransportJob
Research (/) useChat + research stream Long-form briefs, citations, refine, stop/confirm
Voice (/voice) Realtime WebRTC Fast spoken Q&A; phase + transcript + barge-in

Users should feel one app. Engineers keep two pipelines until a later lesson earns a handoff (e.g. "continue this in Research").

What voice must not pretend to be

Reading seven sources aloud fails. Inventing citation numbers aloud fails harder. Session instructions are the product contract: short answers, no fake sources, invite Research mode for depth.

Build it (~20 min)

Four small product moves — shared header, entry point, voice copy, session instructions. Leave the Realtime reducers alone.

  1. Add shared mode chrome. Extract a tiny header (or duplicate the pattern) so both pages show the same product name plus mode links:

    <header className="shrink-0 border-b border-outline-variant/60">
  <div className="mx-auto flex max-w-[1120px] items-center justify-between px-6 py-4">
    <span className="text-base font-semibold tracking-tight">
      Research Assistant
    </span>
    <nav className="flex items-center gap-4 text-sm">
      <Link href="/">Research</Link>
      <Link href="/voice">Voice</Link>
    </nav>
  </div>
</header>

    Put it on the research page header (next to New chat) and wrap the voice page the same way. Active mode can be bold or underlined — keep it quiet.

  2. Add a mic entry from the composer. In ChatComposer, add an icon button that routes to /voice (use next/link or router.push). Tooltip: "Voice mode". Disable it while busy / uploading so you don't abandon an in-flight research turn by accident — or allow it and accept the tradeoff; pick one and stick to it.

    This is the wake/trigger step from voice UX patterns: users shouldn't have to know /voice exists.

  3. Productize the voice surface copy. In VoiceProbe, drop lab language:

    • Eyebrow: remove "Voice probe" (or replace with nothing — the shared header owns the brand).
    • Title: "Voice" or "Talk it through".
    • Dek: something like "Quick spoken answers. Switch to Research for briefs and sources."

    Keep phase badge, transcript, interrupt, latency readout — those are the product, not the scaffolding.

  4. Set research-flavored session instructions. In app/api/realtime/session/route.ts, add instructions on the session mint:

    const secret = await openai.realtime.clientSecrets.create({
  session: {
    type: 'realtime',
    model: 'gpt-realtime-2.1-mini',
    instructions: [
      'You are the voice mode of a research assistant.',
      'Keep answers to 1–3 short spoken sentences.',
      'If the user needs sources, lists, or a written brief,',
      'say so and suggest switching to Research mode.',
      'Never invent citations aloud.',
    ].join(' '),
    audio: { /* unchanged — keep silence_duration_ms: 500 */ },
  },
});

    Reconnect after changing mint config. Ask for a long list out loud — the model should refuse to dump it and point you back to Research.

Watch out

Don't wire Realtime events into useChat this lesson — that's a different architecture (chained or tool-handoff), not a mode switch. Shared header flashing on navigation? Prefer a small shared component over copy-paste drift. Mic button while streaming research? Prefer disabled — abandoning mid-brief feels broken.

Self-check

Why not replace ResearchChat with Realtime speech-to-speech?

Citations, refine-before-send, and structured briefs need durable text and HITL. S2S optimizes for conversational latency, not source-backed research artifacts.

When would chained STT→text→TTS be the better product choice?

When you must reuse the existing research agent unchanged — edit transcript, run tools, approve, then speak. You pay latency for control.

What does "voice for the headline, screen for the data" mean here?

Speak a short summary; put lists, URLs, and citations on the Research surface. Session instructions enforce that split.
