Voice + Image Context

Voice mode answers fast — but users often want to talk about something on screen: a chart, a screenshot, a photo. Today's win: let them attach one image to a live Realtime session, show a thumbnail so they trust the model saw it, then ask aloud.

Why this, why now

Lessons 0010–0016 built a solid voice surface: lifecycle, transcripts, barge-in, latency, mode switch, handoff, turn detection. The voice arc's next product gap is multimodal context — grounding spoken Q&A in what the user is actually looking at.

  • Realtime accepts images as discrete conversation items — not a live video stream.
  • Your app controls when an image enters context; the model doesn't auto-see your screen.
  • For a research assistant, this covers "what trend is in this chart?" before the user switches to Research for a sourced brief.
The core idea

Multimodal voice is explicit attachment + spoken question, not ambient vision. Legibility (thumbnail, disabled-until-connected) matters as much as the API call.

Primary source — read this section (5 min) OpenAI · Realtime conversations — Image inputs — focus on the conversation.item.create example. Cheat sheet: multimodal voice input reference.

Know this first (~4 min)

How images enter a voice session

Send a client event over the WebRTC data channel. The image becomes a user message in conversation history. With VAD on, attaching alone does not trigger a response — the user speaks next, and the model can reference the image when replying.

Why resize on the client

Images travel as base64 inside JSON on the data channel. A full phone photo or uncompressed PNG can be megabytes — slow or brittle. Resize to ~1024 px on the long edge and encode as JPEG before sending. This is a UX requirement, not an optimization nicety.

What voice mode still can't do

Your voice handoff carries transcript text, not images. If the user needs citations or a written analysis of the chart, "Continue in Research" remains the right exit — voice gives a spoken headline; Research gives sources.

Build it (~20 min)

  1. Add a client helper for resize + event payload. Create lib/realtime-image-attach.ts:

    const MAX_EDGE = 1024;
const JPEG_QUALITY = 0.85;

export type PreparedImage = {
  dataUrl: string;
  previewUrl: string;
  label: string;
};

export async function prepareImageFile(file: File): Promise<PreparedImage> {
  if (!file.type.startsWith("image/")) {
    throw new Error("Choose an image file");
  }
  const bitmap = await createImageBitmap(file);
  const scale = Math.min(1, MAX_EDGE / Math.max(bitmap.width, bitmap.height));
  const w = Math.round(bitmap.width * scale);
  const h = Math.round(bitmap.height * scale);
  const canvas = document.createElement("canvas");
  canvas.width = w;
  canvas.height = h;
  canvas.getContext("2d")!.drawImage(bitmap, 0, 0, w, h);
  bitmap.close();
  const dataUrl = canvas.toDataURL("image/jpeg", JPEG_QUALITY);
  return { dataUrl, previewUrl: dataUrl, label: file.name };
}

export function imageAttachEvent(dataUrl: string) {
  return {
    type: "conversation.item.create",
    item: {
      type: "message",
      role: "user",
      content: [{ type: "input_image", image_url: dataUrl }],
    },
  };
}

  2. Wire attach in VoiceProbe. Add state for the pending thumbnail, a hidden file input, and a send helper that uses the open data channel:

    const [attachedImage, setAttachedImage] = useState<PreparedImage | null>(null);

async function attachImage(file: File) {
  const prepared = await prepareImageFile(file);
  const dc = dcRef.current;
  if (dc?.readyState !== "open") throw new Error("Connect first");
  dc.send(JSON.stringify(imageAttachEvent(prepared.dataUrl)));
  setAttachedImage(prepared);
}

    Render an Attach image button (disabled until connected). On success, show a small thumbnail + filename under the phase badge — "Attached — ask about this aloud."

  3. Nudge session instructions. In app/api/realtime/session/route.ts, add one line to the instructions array:

    'When the user attaches an image, describe what you see briefly in spoken form.',
'Do not invent precise numbers from charts unless clearly legible.',

    Reconnect after changing session config.

  4. Run the smoke test.

    • Connect voice mode. Attach a chart or screenshot (PNG/JPEG).
    • Confirm the thumbnail appears before you speak.
    • Ask aloud: "What's the main trend here?" — assistant should reference visible content.
    • Try a 4+ MB original — confirm resize keeps attach snappy.
Watch out

Don't auto-send response.create on attach unless you've disabled VAD — let the user's spoken question drive the turn. Don't promise image carry-over in "Continue in Research" yet; transcript handoff is text-only for v1. Clear the thumbnail when disconnecting so stale context doesn't confuse the next session.

Self-check

Why not stream the webcam continuously?

Realtime image input is discrete messages, not video. Continuous capture burns bandwidth, raises privacy concerns, and your app loses control over what the model sees. Attach-on-demand matches research-style "look at this slide" workflows.

Why show a thumbnail if the model already has the image?

Trust. Users can't inspect session JSON. A visible chip confirms the attach succeeded and sets expectations for the next spoken question.

When should the user switch to Research instead?

When they need numbered sources, a written brief, or editable output — the same boundary as lesson 0014's mode split. Voice describes; Research cites.
