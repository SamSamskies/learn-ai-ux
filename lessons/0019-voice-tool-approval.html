Voice Tool Approval

Lesson 0018 auto-ran a read-only lookup. Side-effecting tools need a gate. Today's win: register stage_research_brief, pause on the function call, show what the model wants to do, and only execute after the user approves.

Why this, why now

Your lookup tool runs the moment the model asks — fine for read-only fetches. But staging a Research draft writes to sessionStorage and changes what the user sees next. That's a side effect: medium harm, medium scope, easy to undo but annoying if accidental.

  • Realtime function tools execute in your client — you choose when to send function_call_output.
  • Hold the output item until approval; the session waits silently unless you label the pause.
  • Split tools by risk: lookup_definition stays auto-run; stage_research_brief gets a confirm dialog.
The core idea

In voice, tool approval is propose → show → decide → resume. The model proposes via function_call; your UI shows the args; approve or reject; then always send output + response.create so speech continues.

Primary source — read this section (5 min) OpenAI · Guardrails and human review — focus on "Approvals are the human-in-the-loop path for tool calls." Pair with Shape of AI · Verification. Cheat sheet: voice tool approval reference.

Know this first (~4 min)

Same loop, different gate

Lesson 0018: response.done → fetch → output → response.create. Today you insert a hold between detect and fetch. The model already stopped speaking; your job is to make the hold legible and show the proposed action.

Rejections must complete the loop too

On Cancel, still send function_call_output with { rejected: true } and response.create. Otherwise the session stalls. Tune instructions so the model acknowledges the decline instead of retrying the tool.

Reuse ConfirmDialog

You already have an accessible modal from lesson 0006. Tool approval is the same component with different copy — show the parsed query in the description so the user approves the actual args, not a vague "Allow action?"

Build it (~20 min)

  1. Add the gated tool + handoff helper. Extend lib/realtime-voice-tools.ts:

    export const STAGE_RESEARCH_BRIEF_TOOL = {
  type: "function" as const,
  name: "stage_research_brief",
  description:
    "When the user wants a sourced research brief, citations, or deep lookup — not a quick definition — stage the topic for Research mode.",
  parameters: {
    type: "object",
    properties: {
      query: {
        type: "string",
        description: "The research topic or question to stage",
      },
    },
    required: ["query"],
  },
};

export const VOICE_TOOLS = [
  LOOKUP_DEFINITION_TOOL,
  STAGE_RESEARCH_BRIEF_TOOL,
] as const;

export function toolApprovalLabel(name: string): string {
  if (name === STAGE_RESEARCH_BRIEF_TOOL.name) {
    return "Waiting for your approval…";
  }
  return "Waiting for approval…";
}

export function rejectedToolOutput(callId: string) {
  return functionOutputEvent(callId, {
    rejected: true,
    reason: "user_declined",
  });
}

    Add to lib/voice-handoff.ts — a formatter that leads with the staged query:

    export function formatStagedResearchHandoff(
  source: VoiceHandoffSource,
  query: string,
): string | null {
  const trimmed = query.trim();
  const context = formatVoiceHandoff(source);
  if (!trimmed && !context) return null;
  const header = trimmed
    ? `Research this topic with sources: ${trimmed}`
    : "Continue this as a source-backed research task.";
  return context ? `${header}\n\n${context}` : header;
}

  2. Register both tools on session mint. In app/api/realtime/session/route.ts, swap to VOICE_TOOLS and extend instructions:

    tools: [...VOICE_TOOLS],
instructions: [
  // …existing lines…
  'Use lookup_definition only for quick term definitions.',
  'Use stage_research_brief when the user wants sources or a written brief.',
  'If stage_research_brief is rejected, acknowledge briefly and offer to stay in voice.',
].join(' '),

  3. Branch tool handling in VoiceProbe. Add pending-approval state and split the handler:

    type PendingToolApproval = {
  call: RealtimeFunctionCall;
  query: string;
};

const [pendingTool, setPendingTool] =
  useState<PendingToolApproval | null>(null);

function handleFunctionCall(call: RealtimeFunctionCall) {
  if (call.name === LOOKUP_DEFINITION_TOOL.name) {
    void runVoiceTool(call);
    return;
  }
  if (call.name === STAGE_RESEARCH_BRIEF_TOOL.name) {
    const query = String(call.args.query ?? "").trim();
    setPendingTool({ call, query });
    setToolLabel(toolApprovalLabel(call.name));
    return;
  }
}

function resolveToolApproval(approved: boolean) {
  const pending = pendingTool;
  if (!pending) return;
  setPendingTool(null);
  setToolLabel(null);

  const dc = dcRef.current;
  if (dc?.readyState !== "open") return;

  if (approved) {
    const draft = formatStagedResearchHandoff(
      { history: transcript.history, draftUser: transcript.draftUser },
      pending.query,
    );
    if (draft) saveVoiceHandoff(draft);
    setTranscript((t) =>
      appendSystemTurn(t, pending.query
        ? `Staged for Research: ${pending.query}`
        : "Staged for Research"),
    );
    dc.send(JSON.stringify(
      functionOutputEvent(pending.call.callId, { staged: true, query: pending.query }),
    ));
  } else {
    dc.send(JSON.stringify(rejectedToolOutput(pending.call.callId)));
  }
  dc.send(JSON.stringify({ type: "response.create" }));
}

    Wire handleFunctionCall where you currently call runVoiceTool directly. Clear pendingTool on disconnect.

  4. Add the approval dialog. Reuse ConfirmDialog — show the query in the description:

    <ConfirmDialog
  open={pendingTool != null}
  title="Stage this for Research?"
  description={
    pendingTool?.query
      ? `This saves a draft and ends voice when you switch: “${pendingTool.query}”`
      : "This saves a research draft you can open in Research mode."
  }
  confirmLabel="Stage draft"
  onConfirm={() => resolveToolApproval(true)}
  onCancel={() => resolveToolApproval(false)}
/>

    Optional: after approve, show a "Open Research" chip that navigates to / — the handoff buffer is already written.

  5. Run the smoke test.

    • Ask "What does VAD mean?" — lookup still auto-runs with Looking up… (no dialog).
    • Ask "Research the history of transformers with sources" — confirm dialog appears with the query; dock shows Waiting for your approval….
    • Approve → assistant confirms aloud; switch to Research → composer seeded with the staged query + transcript context.
    • Repeat, Cancel → assistant acknowledges without staging; no handoff on Research tab.
Watch out

Don't gate read-only lookups — prompt fatigue. Don't skip response.create on reject. Clear pending approval on disconnect/barge-in so stale dialogs don't appear after reconnect. If the model retries rejected tools, tighten the rejection instruction.

Self-check

Why send function_call_output on reject?

The model issued a tool call and is waiting for a result item. Rejection is a valid result — without it, the conversation can't continue to a spoken response.

Why show the query in the dialog?

Voice users may not be looking at the screen. Surfacing parsed args is the voice equivalent of showing a diff before "Apply changes."

What proves the split is working?

Lookup = no dialog, labeled pause. Stage = dialog with query, dock hold label, staged handoff only on approve.
