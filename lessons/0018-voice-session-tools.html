AI Product UX · Phase 2 · Lesson 0018

Voice Session Tools

Voice mode answers from session context — but sometimes it needs a lookup. Today's win: register one Realtime tool, run it when the model asks, show Looking up… while it runs, then let the assistant speak the result.

Why this, why now

Lessons 0010–0017 built lifecycle, transcripts, barge-in, handoff, and image attach. The voice surface still can't do anything outside the model's head. Realtime function calling closes that gap — with a new UX problem: the user stops hearing audio while your code runs.

  • Tools register on the session; the model decides when to call them from spoken intent.
  • A tool turn is response.done → your fetch → function_call_output → response.create — not a normal speaking phase.
  • For a research assistant, keep voice tools lightweight (quick definitions). Sourced briefs stay in Research mode.
The core idea

A voice tool call is a legible pause in the conversation — label it, show what ran, then resume speech. Silent fetches feel like the agent froze.

Primary source — read this section (6 min) OpenAI · Realtime conversations — Function calling — focus on configure → detect function_call → function_call_output → response.create. Cheat sheet: voice session tools reference.

Know this first (~4 min)

Where tools live

Add a tools array when minting the session in app/api/realtime/session/route.ts (same place you set instructions and VAD). Descriptions matter — they steer when the model reaches for a tool vs answering from memory.

Detecting a tool turn

On a normal turn you get audio transcript deltas. On a tool turn, response.done arrives with response.output[0].type === "function_call" and no spoken answer yet. Parse name, arguments (JSON string), and call_id — you need the call id to return results.

The silence gap

Between response.done (function call) and the next audio delta, the user hears nothing. Reuse your thinking phase but change the label — "Looking up…" — and optionally log a system line in the transcript ("Looked up: RAG").

Build it (~20 min)

  1. Define the tool schema + parser. Create lib/realtime-voice-tools.ts:

    export const LOOKUP_DEFINITION_TOOL = {
  type: "function" as const,
  name: "lookup_definition",
  description:
    "Look up a one-sentence definition when the user asks what a term, acronym, or concept means.",
  parameters: {
    type: "object",
    properties: {
      term: {
        type: "string",
        description: "The word or phrase to define",
      },
    },
    required: ["term"],
  },
};

export type RealtimeFunctionCall = {
  callId: string;
  name: string;
  args: Record<string, unknown>;
};

export function extractFunctionCall(event: {
  type: string;
  response?: {
    output?: Array<{
      type?: string;
      name?: string;
      call_id?: string;
      arguments?: string;
    }>;
  };
}): RealtimeFunctionCall | null {
  if (event.type !== "response.done") return null;
  const item = event.response?.output?.find((o) => o.type === "function_call");
  if (!item?.call_id || !item.name) return null;
  let args: Record<string, unknown> = {};
  try {
    args = JSON.parse(item.arguments ?? "{}") as Record<string, unknown>;
  } catch {
    args = {};
  }
  return { callId: item.call_id, name: item.name, args };
}

export function functionOutputEvent(callId: string, output: unknown) {
  return {
    type: "conversation.item.create",
    item: {
      type: "function_call_output",
      call_id: callId,
      output: JSON.stringify(output),
    },
  };
}

  2. Add a stub lookup route. Create app/api/voice/lookup/route.ts — voice tools should stay fast and low-stakes; a canned stub is fine for this lesson:

    import { NextResponse } from "next/server";

const STUBS: Record<string, string> = {
  rag: "Retrieval-augmented generation — fetching external text to ground an answer.",
  vad: "Voice activity detection — deciding when the user has finished speaking.",
};

export async function POST(req: Request) {
  const { term } = (await req.json()) as { term?: string };
  const key = (term ?? "").trim().toLowerCase();
  const definition =
    STUBS[key] ?? `No stub for "${term}" — suggest Research mode for a sourced brief.`;
  return NextResponse.json({ term, definition });
}

  3. Register the tool on session mint. In app/api/realtime/session/route.ts, import LOOKUP_DEFINITION_TOOL and add to the session object:

    tools: [LOOKUP_DEFINITION_TOOL],
tool_choice: "auto",
instructions: [
  // …existing lines…
  'When the user asks what a term means, call lookup_definition instead of guessing.',
  'After a lookup, paraphrase the definition in one short spoken sentence.',
].join(' '),

    Reconnect after changing session config.

  4. Handle the tool loop in VoiceProbe. Track in-flight tool work and extend dc.onmessage:

    const [toolLabel, setToolLabel] = useState<string | null>(null);
const handledCalls = useRef(new Set<string>());

async function runVoiceTool(call: RealtimeFunctionCall) {
  if (call.name !== "lookup_definition") return;
  const term = String(call.args.term ?? "").trim();
  setToolLabel(term ? `Looking up “${term}”…` : "Looking up…");
  const res = await fetch("/api/voice/lookup", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ term }),
  });
  const data = await res.json();
  const dc = dcRef.current;
  if (dc?.readyState !== "open") return;
  dc.send(JSON.stringify(functionOutputEvent(call.callId, data)));
  dc.send(JSON.stringify({ type: "response.create" }));
  setToolLabel(null);
}

// inside dc.onmessage, after phase/transcript reducers:
const call = extractFunctionCall(event);
if (call && !handledCalls.current.has(call.callId)) {
  handledCalls.current.add(call.callId);
  void runVoiceTool(call);
}

    Pass toolLabel into SessionDock — when set, show it instead of "Thinking…". Clear handledCalls on disconnect.

  5. Run the smoke test.

    • Connect voice mode. Ask aloud: "What does RAG mean?"
    • Confirm the dock shows Looking up "RAG"… before speech resumes.
    • Assistant should speak a one-sentence definition grounded in the stub.
    • Ask something that needs sources ("Who invented transformers?") — model should answer briefly or suggest Research, not invent citations.
Watch out

Don't treat tool latency like model thinking — label it. Don't register heavy research tools on voice; they blow the realtime budget and can't show citations. Guard against double-handling the same call_id. If the user disconnects mid-fetch, drop stale response.create sends.

Self-check

Why send response.create after function_call_output?

The function call item only supplies data. The model still needs an explicit response turn to generate the spoken answer using that output.

Why a stub API instead of wiring full Research search?

Voice needs sub-second perceived progress and short answers. Full search + citations belongs in Research mode — same boundary as lessons 0014–0015.

What proves the UX is legible?

During the fetch you see "Looking up…" (not blank silence or generic "Thinking…"), and the spoken answer references the lookup result.
