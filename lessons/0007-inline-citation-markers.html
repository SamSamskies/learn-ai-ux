Inline Citation Markers

Human-in-the-loop gates irreversible actions now. The next trust win is claim-level verification — linking each key point to the source that backs it, not just listing sources at the bottom.

Where you actually are

Your TrustBar tells users how grounded the brief is at scan height. The footer lists source pills. But the synthesis itself is uncited:

Gap — claims disconnected from sources

key_points render as plain facts. A user reading "Revenue grew 12% YoY" has no idea which source supports it until they scroll, cross-reference titles, and guess. That's not verification — it's archaeology.

The Shape of AI citations pattern calls this out: inline cues work for sentence-level claims. Panels and footers work for exploration; markers work for spot-checking.

The principle

Two layers of trust, working together:

Scan height (done)TrustBar — overall confidence, provenance, source count
Claim level (this lesson)Superscript [1] on each cited key point → numbered source below

Numbered markers are the Perplexity pattern: compact inline, full metadata in a list. Multiple markers per claim ([1][2]) signal synthesis across sources.

Primary source (skim) Shape of AI · Citations — inline highlights vs multi-source references, metadata for scanning, and why missing sources must stay explicit. See also inline citations reference.

Build it (~15 min)

  1. Create lib/with-citation-links.ts. Transform model-emitted [1] markers into markdown links the renderer can style:

    const CITATION_RE = /\[(\d+)\]/g;

export function withCitationLinks(text: string): string {
  return text.replace(
    CITATION_RE,
    (_match, n: string) => `[${n}](#source-${n})`
  );
}

  2. Style citation anchors in BriefArticle.tsx — detect #source-N hrefs and render as superscript, skip dead indices:

    function citationComponents(
  sourceCount: number
): Components {
  return {
    ...inlineMarkdownComponents,
    a: ({ href, children }) => {
      const m = href?.match(/^#source-(\d+)$/);
      if (m) {
        const n = Number(m[1]);
        if (n < 1 || n > sourceCount) {
          return <span>[{children}]</span>;
        }
        return (
          <sup className="ml-0.5">
            <a
              href={href}
              className="text-[0.65em] font-semibold text-primary no-underline hover:underline"
              aria-label={`Source ${n}`}
            >
              {children}
            </a>
          </sup>
        );
      }
      // …existing external link styles
    },
  };
}

    Pipe key points through withCitationLinks before ReactMarkdown:

    components={citationComponents(dedupeSources(sources).length)}
>
  {withCitationLinks(point)}
</ReactMarkdown>

  3. Number the source list — add anchor IDs and visible indices so markers have somewhere to land:

    {dedupeSources(sources).map((s, i) => {
  const n = i + 1;
  return (
    <div
      key={sourceKey(s)}
      id={`source-${n}`}
      className="flex scroll-mt-20 items-center gap-2"
    >
      <span
        className="flex h-5 w-5 shrink-0 items-center justify-center rounded-full bg-surface-container-high text-[10px] font-semibold text-on-surface-variant"
        aria-hidden
      >
        {n}
      </span>
      <SourcePill source={s} />
    </div>
  );
})}

  4. Nudge the model — update schema description and system prompt so citations actually appear:

    // lib/schemas.ts — key_points field:
.describe(
  "Three to five concise facts; append [1], [2] etc. when citing retrieved sources (indices match numbered source list)"
)

// lib/stream-research.ts — INSTRUCTIONS, add:
"On key_points, append bracketed source indices like [1] or [1][2] for claims grounded in retrieval. Number sources in the order you rely on them."

  5. Test three paths.

    • Web search question — key points show superscript markers; clicking [1] scrolls to the numbered source
    • Multi-source claim — at least one point with [1][2] if the model synthesizes across sources
    • Broken index — if a marker references a source that doesn't exist, it renders as plain [99] text, not a dead link
What you ship

Claim-level inline citations on key points, bidirectional with a numbered source list. TrustBar still handles scan height; markers handle spot-checking. Next up: edit-before-send on the composer, or a before/after retrofit summary to close the text-surface arc.

Self-check

Why markers on key points, not the summary?

Key points are discrete, verifiable claims — the natural unit for spot-checking. The summary is synthesis; users challenge specific facts first.

Why superscript numbers instead of inline URLs?

URLs break reading flow and look like spam. Numbers are scannable; the source list carries title, favicon-domain context, and the actual link.

Why keep the footer source list if markers are inline?

Markers answer "which source?" — the list answers "what is it?" Users need both: a quick jump and a place to explore.
