Barge-in & Turn-taking UX

Lesson 0002 taught preserve–mark–recover for text stopped. Voice barge-in is the same design move: the server cancels audio, but users still need to see what happened. Today's win: an explicit interrupt button plus interrupted turns marked in your turn log.

Why this, why now

Your probe already handles VAD barge-in at the transport layer — talk over the model and response.cancelled fires. Lesson 0011 already preserves the partial transcript. What's missing is product UX:

  • Interrupted lines look identical to finished ones — users can't tell a cut-off reply from a complete answer.
  • No tap-to-stop — in a noisy room, users won't talk over audio; they expect a button.
  • The phase badge says "Interrupted" globally, but the turn that was cut should carry the mark.

Same three rules as text. Different surface.

The core idea

Interrupt is a turn transition, not an error. Two triggers (VAD speech + explicit cancel), one treatment: preserve partial text, mark the turn interrupted, recover by letting the user keep talking.

Primary source — read this (6 min) OpenAI · Realtime conversations — Interruption and truncation — focus on the WebRTC path (server auto-truncates). Cheat sheet: barge-in reference.

Know this first (~5 min)

Two interrupt paths, one UI

PathWhat happensYour UI job
VAD barge-in User speaks → server cancels response → response.cancelled Show it — phase + transcript. Already mostly wired.
Explicit interrupt User taps Stop → you send response.cancel on the data channel Offer the button while speaking. Same events fire.

Phase is ephemeral; transcript is durable

interrupted on the phase badge is a momentary beat — it clears on the next speech_started. The turn in history needs a permanent interrupted: true flag so users can scroll back and see which replies were cut short.

Build it (~20 min)

Four small changes in VoiceProbe — reducer, transcript render, interrupt button, data-channel ref.

  1. Flag interrupted turns in the reducer. Extend Turn in transcript-reducer.ts:

    export type Turn = {
  role: 'user' | 'assistant';
  text: string;
  interrupted?: boolean;
};

    On response.cancelled, push the partial assistant line with interrupted: true:

    case 'response.cancelled': {
  if (!state.draftAssistant.trim()) return state;
  return {
    history: [
      ...state.history,
      { role: 'assistant', text: state.draftAssistant, interrupted: true },
    ],
    draftUser: state.draftUser,
    draftAssistant: '',
  };
}

    Update the existing test — the assistant turn should include interrupted: true.

  2. Mark interrupted turns in the transcript UI. In Transcript.tsx, when turn.interrupted, render the same badge pattern as text StoppedNotice:

    {turn.interrupted && (
  <span className="mr-2 inline-flex rounded-lg border border-outline-variant bg-surface-container-low px-2 py-0.5 text-xs font-semibold uppercase tracking-wide text-on-surface-variant">
    Interrupted
  </span>
)}
<span className={`whitespace-pre-wrap ${turn.interrupted ? 'text-foreground/60' : ''}`}>
  {turn.text}
</span>

    Muted body text + badge — preserve and mark in one glance. Not red, not an error banner.

  3. Keep a data-channel ref and add an interrupt button. In VoiceProbe/index.tsx:

    const dcRef = useRef<RTCDataChannel | null>(null);

// inside connect(), after createDataChannel:
dcRef.current = dc;

function interrupt() {
  if (dcRef.current?.readyState === 'open') {
    dcRef.current.send(JSON.stringify({ type: 'response.cancel' }));
  }
}

// in resetSession / disconnect:
dcRef.current = null;

    Show the button only while phase === 'speaking', next to Disconnect:

    {connected && phase === 'speaking' && (
  <button type="button" onClick={interrupt}
    className="rounded-lg border border-outline-variant px-5 py-2.5 text-sm font-medium">
    Interrupt
  </button>
)}

  4. Smoke-test both paths.

    • Talk-over: Ask a question that gets a long answer. Speak mid-reply. Phase → interrupted → listening. Turn log shows partial text + Interrupted badge.
    • Tap interrupt: Same long answer, tap the button instead. Same events, same UI treatment.
    • Complete reply: Short answer that finishes naturally — no badge, full-opacity text.
Watch out

Button does nothing? Check dcRef.readyState — must be open. Interrupt badge on every assistant turn? You're setting interrupted on …done — only flag response.cancelled. Phase stuck on interrupted? Confirm speech_started still maps to listening in your phase reducer.

Self-check

Why not show interrupted turns as errors?

Same reason text stopped isn't an error — the user (or their environment) chose to cut the reply. Errors are for things that failed; interrupts are normal turn-taking.

Do you need conversation.item.truncate on WebRTC?

No — the server manages the output audio buffer and truncates unplayed audio automatically on interrupt. Truncation events are for WebSocket clients that own playback.

What's the voice equivalent of regenerate in place?

Just keep talking. VAD opens the floor on the next utterance — no retry button needed on a conversational surface. (Chained STT→text→TTS apps might offer "repeat that" — different architecture.)
